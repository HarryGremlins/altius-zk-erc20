# ECIES（椭圆曲线集成加密方案）

## 概述

在保密转账协议中，链上更新接收方的 commitment 后，接收方需要知道转账金额 `v` 和致盲因子 `r_tx` 才能重建明文余额。问题是：如何通过公开的区块链传递这些秘密数据？

**ECIES** 通过组合椭圆曲线 Diffie-Hellman（ECDH）密钥协商与对称认证加密来解决这个问题。它是一种混合加密方案 — EC 部分建立共享密钥，对称加密算法（AES-GCM）执行实际加密。

---

## 为什么不能直接用 EC 公钥加密？

与 RSA 不同，椭圆曲线密钥**不能直接加密任意数据**。EC 密码学基于离散对数问题，它只提供：

- **签名**（ECDSA）：证明你知道私钥
- **密钥协商**（ECDH）：双方派生共享密钥

没有原生的"用公钥加密消息"操作。ECIES 弥补了这个缺口，将 ECDH 密钥协商转化为完整的加密方案。

---

## ECDH 基础

ECDH（椭圆曲线 Diffie-Hellman）允许双方在不传输共享密钥的情况下派生相同的密钥：

```
Alice 持有：(a, a·G)    — 私钥 a，公钥 a·G
Bob 持有：  (b, b·G)    — 私钥 b，公钥 b·G

Alice 计算：secret = a · (b·G) = ab·G
Bob 计算：  secret = b · (a·G) = ab·G
                     ↑ 相同的点！
```

共享密钥是同一个 EC 点 `ab·G`。任何观察到 `a·G` 和 `b·G` 的第三方都无法计算 `ab·G` — 这就是椭圆曲线 Diffie-Hellman 问题（ECDHP）。

**局限性**：标准 ECDH 使用长期密钥。如果 Alice 用同一对 `(a, b)` 发送多条消息，它们都会派生相同的共享密钥，使方案变成确定性的，容易受到重放攻击。

---

## ECIES：带临时密钥的 ECDH

ECIES 通过让发送方**为每条消息生成新的临时密钥对**来解决确定性问题。这确保每次加密产生唯一的共享密钥。

### 算法（逐步分解）

**准备**：Bob 持有长期密钥对 `(sk, PK)`，其中 `PK = sk·G`。Bob 的 `PK` 是公开的（在我们的系统中注册在链上）。

**加密**（Alice → Bob）：

```
1. 生成临时密钥对
   ek  ← 随机标量                   // 临时私钥（仅使用一次）
   EPK = ek·G                       // 临时公钥

2. ECDH 密钥协商
   S = ek · PK                      // 共享密钥点 = ek · sk · G

3. 密钥派生
   (enc_key, mac_key) = KDF(S)      // 从共享密钥派生对称密钥

4. 对称加密
   ciphertext = AES-GCM(enc_key, plaintext)
   // AES-GCM 同时提供加密和认证（MAC）

5. 输出
   发送：(EPK, ciphertext)          // EPK = ek·G 公开，ek 被丢弃
```

**解密**（Bob）：

```
1. 恢复共享密钥
   S = sk · EPK                     // sk · ek·G = ek · sk·G（同一个点）

2. 密钥派生
   (enc_key, mac_key) = KDF(S)      // 与 Alice 派生的密钥相同

3. 对称解密
   plaintext = AES-GCM.decrypt(enc_key, ciphertext)
   // 同时验证完整性 — 拒绝被篡改的密文
```

### 为什么这能工作

关键洞察是 ECDH 恒等式：

```
Alice 的计算：ek · PK     = ek · (sk · G) = (ek · sk) · G
Bob 的计算：  sk · EPK    = sk · (ek · G) = (sk · ek) · G
                                              ↑ 同一个点
```

双方到达相同的共享密钥 `S`，且从未传输它。窃听者看到 `EPK = ek·G` 和 `PK = sk·G`，但无法计算 `ek·sk·G`（ECDHP）。

---

## ECIES 中的密钥角色

| 密钥 | 所有者 | 生命周期 | 用途 |
|------|-------|---------|------|
| `sk` | 接收方 | 永久 | 长期私钥；用于解密所有收到的消息 |
| `PK = sk·G` | 接收方 | 永久 | 长期公钥；公开发布，供发送方加密 |
| `ek` | 发送方 | 单次使用 | 临时私钥；每条消息新生成，**加密后丢弃** |
| `EPK = ek·G` | 发送方 | 单次使用 | 临时公钥；附在密文中，供接收方恢复共享密钥 |

**发送方在 ECIES 加密中不使用自己的长期密钥。** 临时密钥 `ek` 是发送方的贡献。这意味着：

- 密文**不绑定发送方身份**（默认无发送方认证）
- 每条消息有唯一的共享密钥（即使同一发送方 → 同一接收方）
- 加密完成后泄露 `ek` 不会有影响 — 它已经被丢弃了

---

## 安全属性

### IND-CCA2（密文不可区分性）

攻击者看到 `(EPK, ciphertext)` 后，无法判断加密的是两个候选明文中的哪一个。即使攻击者可以请求解密其他密文（自适应选择密文攻击），该属性依然成立。

临时密钥是实现这一属性的关键 — 即使相同的明文被加密两次发送给同一接收方，两个密文也完全不同，因为每次的 `ek` 都不同。

### 前向保密（部分）

如果 Bob 的长期密钥 `sk` 在未来被泄露，攻击者可以解密所有过去发给 Bob 的 ECIES 消息（因为 `S = sk · EPK` 可以从存储的 `EPK` 值重新计算）。这在接收方密钥维度上**不具有**前向保密性。

但泄露发送方的长期密钥不会影响过去的 ECIES 消息，因为发送方使用的是临时密钥。

### 认证

标准 ECIES **不认证发送方** — 任何知道 Bob 的 `PK` 的人都可以加密消息发给 Bob。在我们的系统中，发送方认证来自区块链交易签名（ECDSA），而非 ECIES 本身。

---

## ECIES 与其他方案对比

| 维度 | ECIES | RSA 加密 | 纯对称加密（预共享密钥） |
|------|-------|---------|----------------------|
| 密钥类型 | 椭圆曲线 | RSA 密钥对 | 预共享对称密钥 |
| 工作方式 | ECDH + AES 混合 | RSA-OAEP 直接加密 | 用共享密钥做 AES-GCM |
| 128 位安全性所需密钥长度 | 256-bit EC | 3072-bit RSA | 128-bit AES |
| 密文额外开销 | ~33 字节（压缩 EPK）+ AES 开销 | ≥384 字节（RSA-3072） | 仅 AES 开销 |
| 每条消息唯一密钥 | ✅ 临时密钥 | ✅ 随机填充 | ❌ 复用同一密钥 |
| 链上适用性 | ✅ 紧凑 | ❌ 密钥和密文都很大 | ❌ 需要密钥交换通道 |

## 在本项目中的应用

在 Altius 隐私模块中，ECIES 用于传递加密转账备忘录：

```
保密转账：Alice → Bob

1. Alice 打包转账元数据：plaintext = (v, r_tx)
   v:    转账金额
   r_tx: 转账 commitment Com(v, r_tx) 的致盲因子

2. Alice 用 ECIES 加密：
   - Bob 的 PK = PK_ecies_bob（从链上注册表查找）
   - ek ← 新鲜随机标量
   - S = ek · PK_ecies_bob
   - sym_key = KDF(S)
   - ciphertext = AES-GCM(sym_key, [v, r_tx])
   - 将 (ek·G, ciphertext) 作为 calldata 附加到交易中

3. Bob 解密：
   - S = sk_ecies_bob · ek·G
   - sym_key = KDF(S)
   - [v, r_tx] = AES-GCM.decrypt(sym_key, ciphertext)
   - 更新本地状态：balance_new = balance_old + v, r_new = r_old + r_tx
```

### 密钥映射到本系统

| ECIES 通用名称 | 本系统中的名称 | 说明 |
|---------------|-------------|------|
| 接收方 `sk` | `sk_ecies` | 从 EVM 密钥派生：`KDF("ecies" \|\| evm_sk)` |
| 接收方 `PK` | `PK_ecies` | 通过 `register()` 注册在链上 |
| 发送方 `ek` | `ek` | 每笔交易新生成，加密后丢弃 |
| 临时公钥 | `ek·G` | 存储在交易 calldata 中（压缩后约 33 字节） |
| 共享密钥 | `secret` | `ek · PK_ecies_receiver` = `sk_ecies_receiver · ek·G` |
| 对称密钥 | `sym_key` | `KDF(secret)` |
| 明文 | `[v, r_tx]` | 转账金额和致盲因子 |

### 链上开销

每笔保密转账增加约 80 字节 calldata：
- `ek·G`：33 字节（压缩 EC 点）
- AES-GCM 密文：约 48 字节（16 字节数据 + 12 字节 nonce + 16 字节认证标签）

### 为什么选择 ECIES 而非更简单的方案？

1. **无需预共享密钥**：发送方只需接收方的公钥（链上可查）
2. **紧凑**：约 80 字节开销，RSA 需要数百字节
3. **标准构造**：经过充分分析，被 Ethereum devp2p（RLPx）、SECG SEC 1、IEEE 1363a 采用
4. **适配账户模型**：每个用户注册一个 ECIES 公钥，任何发送方都可以加密发给任何已注册的接收方
